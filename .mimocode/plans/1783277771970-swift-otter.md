# Task Tracker Feature — Implementation Plan

## Overview

Add a Task Tracker feature to the existing obsidian-calendar-plugin-remastered. The feature adds a collapsible task panel below the calendar in the right sidebar, with full CRUD for tasks and projects, calendar integration via colored dots, drag-and-drop date reassignment, and optional note-backed tasks.

### 1.2 Storage Service

**File: `src/task-tracker/storage.ts`** (~120 lines)

Purpose: Read/write `tasks.json` and `projects.json` via Obsidian's `Plugin.loadData()` / `Plugin.saveData()` — or direct vault API for the data files.

The plugin already uses `this.loadData()` / `this.saveData()` in `src/main.ts` which reads/writes `data.json`. For the task tracker, we store tasks and projects in **separate files** under the plugin's data directory using the vault API, to avoid bloating `data.json` and to keep concerns separated.

However, since the plugin doesn't currently have access to a dedicated data folder, the simplest approach that matches existing patterns is to store everything in `data.json` under new top-level keys:

```json
{
  "wordsPerDot": 250,
  "weekStart": "locale",
  // ... existing settings ...
  "taskTracker": {
    "tasks": [],
    "projects": [],
    "version": 1
  }
}
```

Key exports:

```typescript

export async function saveTaskData(plugin: CalendarPlugin, data: ITaskTrackerData): Promise<void>;
export function generateId(): string;  // crypto.randomUUID or fallback
```

## Phase 3: Calendar Integration

### 3.1 Task Dot Source (ICalendarSource)

**File: `src/task-tracker/taskDotSource.ts`** (~70 lines)

Purpose: New `ICalendarSource` implementation that shows colored dots on calendar days based on tasks assigned to that date, colored by project.

Key pattern (follows existing `tasksSource` in `src/ui/sources/tasks.ts`):

```typescript
import type { Moment } from "moment";
import type { ICalendarSource, IDayMetadata, IDot } from "obsidian-calendar-ui";
import { get } from "svelte/store";
import { tasks, projects } from "./stores";
import { getDateUID } from "obsidian-daily-notes-interface";

function getDotsForDate(dateUID: string): IDot[] {
  const allTasks = get(tasks);
  const allProjects = get(projects);
  const dateTasks = allTasks.filter((t) => t.dateUID === dateUID && !t.completed);

  // Group by project, return one dot per project (max ~4 dots)
  const projectMap = new Map<string, number>();
  for (const task of dateTasks) {
    const pid = task.projectId || "_none";
    projectMap.set(pid, (projectMap.get(pid) || 0) + 1);
  }

  const dots: IDot[] = [];
  for (const [pid] of projectMap) {
    const project = allProjects.find((p) => p.id === pid);
    dots.push({
      color: project?.color || "default",
      className: "task-tracker-dot",
      isFilled: true,
    });
  }
  return dots.slice(0, 5);  // max 5 dots per day
}

export const taskDotSource: ICalendarSource = {
  getDailyMetadata: async (date: Moment): Promise<IDayMetadata> => {
    const dateUID = getDateUID(date, "day");
    return { dots: getDotsForDate(dateUID) };
  },
  getWeeklyMetadata: async (date: Moment): Promise<IDayMetadata> => {
    const dateUID = getDateUID(date, "week");
    return { dots: getDotsForDate(dateUID) };
  },
};
```

Dependencies: `obsidian-calendar-ui` (ICalendarSource, IDayMetadata, IDot), `src/task-tracker/stores.ts`, `obsidian-daily-notes-interface` (getDateUID).

### 4.2 Drop Targets on Calendar Days

**File: Modify `src/view.ts`** (register drop targets) OR create **`src/task-tracker/dateDropTarget.ts`** (~50 lines)

The calendar UI (`obsidian-calendar-ui`) renders day cells. We need to make those cells accept drops. Two approaches:

**Approach A (Recommended): Use Obsidian's native drag-drop on the view**
Since `CalendarView` renders into `contentEl`, we can add a global drop listener on `contentEl`:

```typescript
// In CalendarView.onOpen(), after creating the Calendar Svelte component:
(this as any).contentEl.addEventListener("dragover", (e: DragEvent) => {
  e.preventDefault();
  e.dataTransfer.dropEffect = "move";
});

(this as any).contentEl.addEventListener("drop", (e: DragEvent) => {
  e.preventDefault();
  const taskId = e.dataTransfer.getData("text/plain");
  if (!taskId) return;

  // Determine which date cell was dropped on by walking up from e.target
  const dayEl = (e.target as HTMLElement).closest("[data-date]");
  if (!dayEl) return;
  const dateUID = dayEl.getAttribute("data-date");
  if (dateUID) {
    moveTask(taskId, dateUID);
  }
});
```

This relies on `obsidian-calendar-ui` setting `data-date` attributes on day cells. If it doesn't, we fall back to coordinate-based detection.

**Approach B: Svelte-based drop zones**
Create a `DropZone.svelte` wrapper that wraps each task item and calendar cell. More complex but more Svelte-idiomatic.

Dependencies: `src/task-tracker/stores.ts` (moveTask), `src/view.ts`.

```typescript
export async function createNoteTask(
  task: ITask,
  project: IProject | null,
  app: App
): Promise<TFile> {
  const folder = project?.folder || "";
  const filename = sanitizeFilename(task.title) + ".md";
  const path = folder ? `${folder}/${filename}` : filename;

  const content = [
    "---",
    `task_id: ${task.id}`,
    `project: ${project?.name || ""}`,
    `completed: ${task.completed}`,
    `date: ${task.dateUID}`,
    "---",
    "",
    `# ${task.title}`,
    "",
    "Start writing here...",
  ].join("\n");

  const file = await app.vault.create(path, content);
  return file;
}
```

### 5.2 Frontmatter Synchronization

**File: `src/task-tracker/noteTasks.ts`** (continued)

Listen for vault `modify` events to sync frontmatter changes back to the task store:

```typescript
export function setupNoteTaskSync(app: App): void {
  app.vault.on("modify", async (file: TFile) => {
    if (!(file instanceof TFile)) return;
    const cache = app.metadataCache.getFileCache(file);
    const frontmatter = cache?.frontmatter;
    if (!frontmatter?.task_id) return;

    const taskId = frontmatter.task_id;
    const allTasks = get(tasks);
    const task = allTasks.find((t) => t.id === taskId);
    if (!task) return;

    // Sync from file → store
    updateTask(taskId, {
      title: frontmatter.title || file.basename,
      completed: frontmatter.completed ?? task.completed,
      dateUID: frontmatter.date ?? task.dateUID,
    });
  });
}
```

Call `setupNoteTaskSync(app)` from `main.ts.onload()`.

## Key Architectural Decisions

1. **Storage in `data.json`** rather than separate vault files — matches existing plugin pattern, avoids vault permissions complexity, keeps data with the plugin lifecycle.

2. **Svelte stores for state** — matches existing `src/ui/stores.ts` pattern exactly. No new state management library needed.

3. **Obsidian Modal for dialogs** (not Svelte) — matches existing `ConfirmationModal` pattern. Lower learning curve, native Obsidian look and feel.

4. **ICalendarSource for dots** — the calendar already supports multiple sources. Adding a new source is the cleanest integration path and requires zero changes to `obsidian-calendar-ui`.

5. **HTML5 drag-and-drop** — no library needed. Works on desktop and mobile (with touch event polyfill). The calendar already renders day cells we can attach drop targets to.

6. **`selectedDate` as shared store** — bridges calendar clicks to the task panel without tight coupling. The calendar doesn't need to know about the task panel; it just updates the store.

7. **Note tasks as opt-in** — JSON tasks are the default (simpler, faster). Note tasks are a power-user feature for linking tasks to actual notes with frontmatter metadata.

## Scope Decision

**Full implementation** — all 7 phases, implemented as an extension of the existing Calendar Plugin Remastered (not a separate plugin). The task panel renders below the calendar in the same right sidebar view.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| `obsidian-calendar-ui` day cells may not have `data-date` attributes for drop targets | Fall back to element position detection or add drop targets via `MutationObserver` |
| Task panel height may push calendar out of view in narrow sidebars | Use `max-height: 50vh` with overflow scroll; add collapse toggle |
| `data.json` bloat with many tasks | Add a warning when task count exceeds 1000; suggest archiving |
| Note frontmatter sync race conditions | Debounce store→file sync by 500ms; use file mtime to detect conflicts |
| Svelte 3.35 has no `use:` action for drag-drop | Use inline `on:dragstart` handlers instead of actions |
